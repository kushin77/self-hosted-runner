## 🏗️ Allowed Operations (Safe Without Asking)

These operations are safe and don't require permission:

- `git status` - Check branch/file status
- `git log` - View commit history
- `git diff` - Review changes
- `git branch -a` - List branches
- `git tag` - List tags
- `git show` - Display commit info
- Reading files (no git operations)
- Planning work (no git operations)

## 📋 Required Branch Naming Pattern

**Format:** `TYPE/TICKET-NUMBER-description`

**Valid Types:**
- `feature/` - New features
- `fix/` - Bug fixes  
- `docs/` - Documentation only
- `refactor/` - Code refactoring
- `perf/` - Performance improvement
- `ci/` - CI/CD changes
- `test/` - Test additions
- `chore/` - Maintenance tasks
- `security/` - Security patches

**Examples:**
```
✅ CORRECT
- feature/INFRA-401-add-oauth-support
- fix/AUTH-523-expired-token-handling
- docs/improving-api-documentation
- security/CVE-2026-0001-fix

❌ WRONG
- feature-add-oauth (missing ticket)
- INFRA-401 (missing type prefix)
- random-branch-name (invalid format)
- adding-stuff (too vague)
```

## 💬 Commit Message Format

**Format:** `type(scope): message [#TICKET]`

**Required Structure:**
```
type: One of [feat, fix, docs, refactor, perf, ci, test, chore, security]
scope: Module/component affected (optional but recommended)
message: Clear, concise description
ticket: Issue/ticket number (optional but encouraged)
```

**Examples:**
```
✅ CORRECT
- feat(auth): add OAuth2 support #1234
- fix(api): handle expired tokens gracefully #1235
- docs: update API documentation
- security: CVE-2026-0001 token validation

❌ WRONG
- added feature
- wip
- stuff
- asdf
```

## 🔐 Security Rules

**Secrets:**
- Never stage files containing passwords, keys, tokens
- Use `git-secrets` pre-commit hook (installed)
- Run `git secrets --scan` before adding files

**Signing:**
- All commits to main must be signed (`git config commit.gpgSign true`)
- Only release commits allowed unsigned on release branches

**Access:**
- Never push with personal tokens
- Use GitHub-provided tokens only
- Never store credentials in environment variables

## 📤 Push Operations

**BEFORE any push, execute:**
```bash
git fetch origin              # Get latest commits
git log origin/BRANCH..HEAD   # Show what you're pushing
git diff origin/BRANCH HEAD   # Show changes
```

**NEVER push if:**
- Branch is behind origin (run git pull first)
- Unresolved conflicts exist
- Uncommitted files in working directory
- Last commit is unsigned
- Branch name is invalid

## 🔄 Pull/Merge Workflow

**Correct flow:**
```
1. Create feature branch: git checkout -b feature/TICKET-description
2. Work: git add . && git commit -m "type: message"
3. Push: git push origin feature/TICKET-description
4. Create PR (use GitHub UI)
5. Wait for review + approval
6. Use "Squash and merge" only
7. Delete branch after merge
```

**Never do:**
- Direct push to main
- Merge without PR
- Force-merge
- Skip code review
- Commit directly to protected branches

## ⚠️ Error Handling

**If an error occurs:**

1. **STOP immediately**
2. **DO NOT retry**
3. **Ask user for guidance**
4. **Show full error output**
5. **Suggest manual recovery steps**

**Example:**
```
❌ Command failed: git push origin main
Error: "refused: updates would be overwritten by remote merge"

This means:
- Origin/main has commits we don't have
- We need to pull first

Manual fix:
git fetch origin
git rebase origin/main
git push origin main
```

## 🔄 Common Workflows

### Adding a Feature

```bash
# 1. Create branch
git checkout -b feature/INFRA-401-oauth

# 2. Work locally (commit as needed)
git add file.ts
git commit -m "feat(auth): add OAuth2 provider support #401"

# 3. ASK FIRST
"Ready to push feature/INFRA-401-oauth? YES/NO"

# 4. Show work
git log -1
git diff origin/main

# 5. WAIT for YES

# 6. Push
git push origin feature/INFRA-401-oauth

# 7. Create PR (GitHub UI)
# 8. Wait for review
# 9. Squash and merge (not rebase/merge)
```

### Fixing a Bug

```bash
# Create fix branch
git checkout -b fix/AUTH-523-expired-token

# Fix + commit
git commit -m "fix(api): handle expired tokens {#523}"

# ASK + Show + WAIT + Execute
# (same as feature workflow above)
```

### Emergency Hotfix

```bash
# 1. Create hotfix branch FROM MAIN
git checkout main
git pull origin main
git checkout -b hotfix/URGENT-critical-bug

# 2. Fix + commit
git commit -m "fix(critical): patch urgent issue"

# 3. ASK user
# 4. Push to hotfix branch
# 5. Create PR requesting urgent merge
# 6. Wait for immediate review (30-min SLA)
# 7. Merge to main + staging (marked urgent)
```

## 📝 Special Cases

### Reverting a Commit

```
git revert COMMIT_SHA  # Creates new commit that undoes changes
git push origin BRANCH  # Safe! Never changes history
```

### Squashing Local Commits Before Push

```
git rebase -i origin/main  # ONLY on feature branches
# Mark commits as "squash"
# Push after
```

### Updating Feature Branch

```
git fetch origin
git rebase origin/main  # On feature branches only
git push origin BRANCH
```

### Deleting Feature Branch

```
git branch -d feature/TICKET  # After merged to main
git push origin --delete feature/TICKET  # Remote delete
```
