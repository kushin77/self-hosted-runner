### Path 2: Local Vault + AWS IAM Keys (Self-Contained)

**What You Need to Provide**:
1. AWS Access Key ID (starts with `AKIA...`, 20 chars)
2. AWS Secret Access Key (40 chars)
3. Optional: AWS Session Token (for temporary STS credentials)

**Recommended**: Use **temporary STS credentials** (60–3600 sec TTL), not long-lived IAM user keys.

**How to Generate STS Credentials** (if you have AWS CLI access):
```bash
aws sts get-session-token --duration-seconds 3600
# Returns:
# {
#   "Credentials": {
#     "AccessKeyId": "ASIA...",
#     "SecretAccessKey": "...",
#     "SessionToken": "..."
#   }
# }
```

**What Happens Automatically** (agent runs these):
```bash
# Environment variables set from provided credentials
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="..."

# Configure local Vault AWS secrets engine
sudo VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=$(cat /root/vault_root_token) \
  /usr/local/bin/vault secrets enable -path=aws aws || true

sudo VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=$(cat /root/vault_root_token) \
  /usr/local/bin/vault write aws/config/root \
    access_key="$AWS_ACCESS_KEY_ID" \
    secret_key="$AWS_SECRET_ACCESS_KEY" \
    region=us-east-1

# Create role for temporary IAM user generation
sudo VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=$(cat /root/vault_root_token) \
  /usr/local/bin/vault write aws/roles/nexusshield-role \
    credential_type=iam_user \
    policy_arns='arn:aws:iam::aws:policy/ReadOnlyAccess'

# Restart agent to re-render templates with real credentials
sudo systemctl restart vault-agent.service

# AWS inventory commands run
source /var/run/secrets/aws-credentials.env
aws sts get-caller-identity > cloud-inventory/aws-sts-identity.json
aws s3api list-buckets > cloud-inventory/aws-s3-buckets.json
aws ec2 describe-instances --region us-east-1 > cloud-inventory/aws-ec2-instances.json
aws rds describe-db-instances > cloud-inventory/aws-rds-instances.json
aws iam list-users > cloud-inventory/aws-iam-users.json
```

**Timeline**: 10–15 minutes from credential provision  
**Security**: ✅ If using STS temp creds (TTL 1 hour); credentials stored in local Vault (lost on bastion restart)  
**Caveat**: Local Vault is ephemeral; requires re-initialization after bastion restart

## 🔒 Security Notes

### Local Vault (Current State)
- File-based storage at `/var/lib/vault/data` (owned by root, mode 0700)
- Root token in `/root/vault_root_token` (mode 0600, readable only by root)
- AppRole credentials in `/var/run/vault/` (ephemeral, lost on reboot)
- No backup/replication; if bastion restarts, must re-initialize

### Production Vault Path (Recommended)
- Uses short-lived admin token (1-hour TTL) — restricted exposure window
- AppRole credentials already on bastion (no new secrets needed)
- Production Vault handles audit logging, HA, and backup

### AWS Credentials Best Practices
- **Prefer**: Temporary STS credentials with 1–3600 second TTL (not long-lived IAM keys)
- **Scope**: Least-privilege IAM policy (ReadOnlyAccess for inventory use)
- **Rotation**: Vault can auto-rotate (if configured in AWS secrets engine)
- **Audit**: All API calls logged to CloudTrail
