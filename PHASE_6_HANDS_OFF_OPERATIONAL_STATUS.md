## Automation System Architecture

### 1. GCP Key Fallback Provisioning (`gcp-key-fallback-provisioning.yml`)
**Purpose**: Automatically provision GCP credentials if missing  
**Schedule**: Every 6 hours + Manual trigger  
**Behavior**:
- ✅ Check if `GCP_SERVICE_ACCOUNT_KEY` exists
- ✅ If missing: Auto-provision test key
- ✅ If environment key present: Use it (preferred)
- ✅ Trigger cascade on successful provision
- **Idempotency**: Safe to run multiple times

**Flow**:
```
Check Key → Missing? → Provision Fallback → Trigger Cascade
         → Exists? → Skip (noop)
```

### 2. Auto-Activation Cascade (`auto-activation-cascade.yml`)
**Purpose**: Automatically detect ready state and trigger full cascade  
**Schedule**: Every 5 minutes + Manual trigger  
**Behavior**:
- ✅ Check all 3 critical secrets (GCP, SSH, Runner token)
- ✅ When ALL ready: Auto-trigger verification + DR tests
- ✅ Post progress to Issue #1239
- ✅ Auto-close issue on success
- **Idempotency**: Skips if workflows already running

**Flow**:
```
Check Secrets → All Present? → YES → Dispatch Pipelines → Auto-close #1239
                             → NO  → Wait 5 min & Retry
```

### 3. Self-Healing Remediation (`self-healing-remediation.yml`)
**Purpose**: Auto-detect and fix common failure patterns  
**Schedule**: Every 10 minutes + Manual trigger  
**Behavior**:
- ✅ Detect missing secrets → Auto-provision fallbacks
- ✅ Detect failed workflows → Auto-retry with exponential backoff
- ✅ Generate diagnostic reports
- **Idempotency**: Skips already-provisioned secrets

**Flow**:
```
Detect Issue → Fallback Available? → YES → Provision & Retry
            → NO → Log & Alert
```

- `dr-smoke-test.yml` - Disaster recovery validation
- `operational-health-dashboard.yml` - Real-time metrics

## Secret Provisioning Hierarchy

### Tier 1: Production (Preferred)
- **GCP**: From environment `$GCP_SERVICE_ACCOUNT_KEY`
- **SSH**: From environment `$DEPLOY_SSH_KEY`  
- **Token**: From environment `$RUNNER_MGMT_TOKEN`
- **Source**: Operator/Admin provision
- **Frequency**: One-time (on demand)

### Tier 2: Test Keys (Fallback)
- **GCP**: Minimal valid JSON structure (test-project)
- **SSH**: Test PEM key (valid format, test only)
- **Token**: Generated test token (ghr_test_*)
- **Source**: Auto-generated by fallback provisioning
- **Frequency**: Every 6 hours if production unavailable
- **Purpose**: Keep system operational for development/testing

### Tier 3: Self-Healing (Recovery)
- **Detection**: Every 10 minutes
- **Action**: Provision Tier 2 if missing
- **Recovery**: Attempt workflow retries automatically
- **Purpose**: Zero-downtime resilience

**Expected**: Workflows dispatch within 2 minutes


**Auto-Fix**: Every 10 minutes (self-healing retries)
**Manual Fix**:
```bash
gh workflow run self-healing-remediation.yml --ref main
```
**Expected**: Auto-retry cascade initiated

### Issue: Test Keys in Production
**Fix**: Provision production secrets via environment
**Process**: 
1. Set `GCP_SERVICE_ACCOUNT_KEY` environment variable

3. Set `RUNNER_MGMT_TOKEN` environment variable
4. Trigger: `gh workflow run gcp-key-fallback-provisioning.yml --ref main`
**Result**: Production secrets replace test keys
