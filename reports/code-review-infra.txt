Static Infrastructure & Deployment Review
======================================

Scanned files:
- scripts/automation/pmo/deploy-full-stack.sh
- ansible/inventory/staging
- ansible/playbooks/provision-self-hosted-runner.yml
- scripts/automation/pmo/systemd/actions-runner.service
- scripts/automation/pmo/nuke-and-restore.sh (new)

Findings (high-level):

1) Hard-coded defaults and inconsistent SSH user
   - `deploy-full-stack.sh` has multiple user defaults (`akushnir`, `cloud`). Prefer single source of truth and environment-driven configuration.

2) Lack of idempotency and containerization
   - Services are started via `nohup node ...` and `http-server` rather than container images or systemd units. Hard to guarantee immutability & reproducibility.

3) Limited error handling and monitoring
   - Scripts often `pkill` and `nohup` without supervising processes; no restart policies or health checks beyond simple curl.

4) Secrets & tokens handling
   - Ansible playbook requires `runner_registration_token` but doesn't provide an automated vault lookup. Sensitive values must be stored in Vault or encrypted Ansible vault.

5) No verified atomic restore path
   - Current deploy script copies files in place and starts processes. There's no atomic switch-over (e.g., symlink-based release directory, container image rollouts, or systemd transient units).

6) Missing TLS and minimal network hardening
   - Portal served via `http-server` on 0.0.0.0; no TLS termination or firewall configuration in automation.

7) Ports and host assumptions fixed in script
   - Ports 3919, 9090, 3000 are assumed free; deploy script doesn't validate or set firewall rules.

Recommendations (short):
- Parameterize `TARGET_USER` and `TARGET_HOST` from environment and shared config.
- Adopt container images (Docker/OCI) and orchestrate with systemd + podman-compose, docker-compose, or Kubernetes for immutability.
- Use Ansible roles with idempotent tasks; store secrets in Vault or Ansible Vault.
- Implement release directories and symlink switch (e.g., /opt/apps/releases/<ts> -> /opt/apps/current) for atomic rollbacks.
- Add health checks, systemd units (or containers with restart policies), and monitoring endpoints.
- Bake images with CI and store artifacts in a registry for reproducible deploys.

This file is an automated static review summary created as a first pass; follow-ups should include deeper code security scan and dependency SBOM generation.
